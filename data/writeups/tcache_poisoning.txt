Exploiting BabyHeap with Tcache Poisoning

We are given a heap pwn challenge with a typical menu binary.
The binary has four functions: allocate, delete, view, and update.
We can allocate up to 16 chunks with a fixed size of 0x68 bytes.
The vulnerability is in the delete function: it frees a chunk but does not NULL out the pointer.
This gives us a use-after-free (UAF) and double-free primitive.

First, we need to leak the heap base address to bypass Safe Linking.
Glibc 2.34+ introduced Safe Linking which XORs the tcache fd pointer with (chunk_addr >> 12).
When we double free a chunk and view its contents, we can read the mangled fd pointer.
The mangled fd value is: ptr XOR (addr >> 12).
Since ptr is NULL for the first free in tcache (the fd points to NULL after first free),
we get: 0 XOR (addr >> 12) = (addr >> 12).
So by reading the fd, we get the XOR key, which is heap_base >> 12.
This gives us the heap base address by left-shifting 12 bits.

Next, we overwrite the fd to poison the tcache.
We use the UAF to edit the freed chunk's fd.
We want to allocate a chunk at a target address like __malloc_hook or a GOT entry.
But first we need to leak libc base address.

To leak libc, we need a chunk in the unsorted bin.
The unsorted bin contains libc pointers when a free chunk is the only one in the bin.
We can forge a chunk size larger than the tcache max (0x410) so that when freed, it goes to unsorted bin.
We overwrite the next chunk's size field to create a fake large chunk.
Then we free that fake chunk to get an unsorted bin leak.

With libc base known, we compute __malloc_hook and __free_hook addresses.
We can then poison the tcache fd to point to __free_hook or __malloc_hook.
After allocating at that target, we overwrite it with system address.
Then we trigger a free with /bin/sh as argument to spawn a shell.

We also need to bypass the tcache double-free detection in glibc 2.34+.
The tcache has a "key" field in the freed chunk that points to the tcache_perthread_struct.
If we try to double free without clearing this key, glibc detects it.
We can bypass this by first editing the freed chunk to overwrite the key with NULL,
then free again to trigger the double free.
After that, we edit the fd to poison the tcache bin.

The full exploit sequence is:
1. Allocate three chunks (chunk0, chunk1, chunk2)
2. Free chunk0 twice (with key bypass) to get a double free in tcache
3. Edit chunk0 to poison fd with target address
4. Allocate to pop the poisoned chunk
5. Allocate again to land at our target address
6. Overwrite with system or one_gadget
7. Trigger the overwritten function pointer to get shell

The key is to carefully manipulate the tcache bin state and bypass the safety checks.
Safe Linking protection XORs the fd with (chunk_addr >> 12), so we must use the leaked XOR key.
